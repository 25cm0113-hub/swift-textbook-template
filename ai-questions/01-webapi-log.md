## 質問と回答の記録

## Q1
質問： .addingPercentEncodingというメソッドってどういう意味？

AIの回答の要点： URLでは、使えない文字（日本語・スペース・記号など）があります。それらを % を使った形式に変換するのが「percent encoding（パーセントエンコーディング）」です。


(例) 日本語 → %E3%81%93... のように変換されるスペース → %20 URLに含めても壊れないようにするための処理

自分の理解： URLには日本語やスペースなど使えない文字があるので今回の.addingPercentEncodingメソッドを使うことによって %〇〇などのURLに使えるようにして日本語などでも検索できるようにしている。


## Q2
質問： なぜ検索ワードをURLに入れる際に.urlQueryAllowedで指定するの？

AIの回答の要点： クエリ(検索ワード)にはルールがある👇

= → キーと値の区切り & → パラメータの区切り英数字 → OK スペース → NG（エンコード必要）

👉 .urlQueryAllowed はこのルールに合わせて「残していい文字」が調整されてる

自分の理解： URLには直接日本語を入れることができないので.urlQueryAllowedで指定することによってクエリにいい感じに調整して入れることができるようになっている

## Q3

質問：なぜわざわざ非同期処理で処理を行う？

AIの回答の要点：
■ await を使わないとどうなるか
例えばネット通信（API取得）やファイル読み込みは時間がかかる。
普通にやると
→ 終わるまで処理が止まる（同期処理）
つまり
画面が固まる
次の処理に進めない

■ await を使うとどうなるか
await は「終わるまで待つ」けど、
アプリ全体は止めないという動きになる。
イメージ：
処理を裏で進める
結果が来たらそこだけ再開する

自分の理解：
通信やファイル操作は比較的に重い処理なのでアプリのUIを固まらせないように裏で非同期処理しないといけない.
しないと思いアプリとなって意しまう可能性がある.



## Q4

質問 : 構造体のIdentifiableって?

AIの解答の要件:
Identifiable は 「このデータを一意に識別できますよ」 ということを SwiftUI に伝えるためのプロトコルです。


例：
ForEach(songs) { song in
    Text(song.trackName)
}

のように ForEach で使えるようになります。

質問 : 上の構造体では使わないのはなぜ？

SearchResponse が Identifiable を使っていないのは、一覧表示で1件ずつ扱うデータではないからです。


## Q6

質問 : Cordbleとは

AIの解答の要件:

Codable は データを変換できるようにするプロトコル です。

主に
JSON → Swiftの構造体（デコード）
Swiftの構造体 → JSON（エンコード）
に使います。

Decodable → JSONから構造体へ変換
Encodable → 構造体からJSONへ変換
Codable → 両方できる
